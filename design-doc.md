## Problem Statement

FT Visual Story 수준의 인터랙티브 비주얼 스토리텔링은 현재 전담 개발자+디자이너 팀이 수주에 걸쳐 제작하며, 건당 수천만원의 비용이 든다. 이로 인해 1인 미디어, 뉴스레터 운영자, 소규모 콘텐츠 크리에이터는 이 포맷에 접근조차 할 수 없다. AI 멀티에이전트 시스템으로 "토픽 → 완성된 비주얼 스토리"를 자동화하면, 이 비용 장벽을 근본적으로 부술 수 있다.

## Demand Evidence

- **검증된 외부 수요: 없음** — 아직 프리프로덕트 단계
- **파운더 자신의 통증:** KP 본인이 콘텐츠 크리에이터로서 FT급 비주얼 콘텐츠를 만들고 싶지만, 기술/비용 장벽이 높아서 못 만들고 있음
- **기술적 확신:** ebook-writer 프로젝트에서 "토픽 → 리서치 → 구조 → 집필 → 편집 → 이미지 → 최종 산출물" 전체 파이프라인 자동화를 이미 증명함
- **시장 갭:** Shorthand($399/월)은 레이아웃만 쉽게 해주고 콘텐츠는 사람이 만들어야 함. AI 뉴스레터 도구(beehiiv, Piktochart)는 이메일 레이아웃에 집중하며 FT급 롱폼과는 거리가 멀다. **토픽 → FT Visual Story 전체를 자동 생성하는 도구는 현재 존재하지 않음**

## Status Quo

| 유저 유형 | 현재 방식 | 비용 | 품질 |
|----------|----------|------|------|
| FT/NYT 미디어 | 전담 개발자+디자이너 팀 | 수천만원/건 | 최고 |
| 대기업 마케팅 | 에이전시 외주 | 수백만원/건 | 높음 |
| 스타트업 | Canva/Notion으로 타협 | $13/월 | 낮음 |
| 1인 미디어 | Substack 텍스트 only | $0 | 없음 |
| 컨설턴트 | PPT + PDF | 노동시간 | 중간 |

**핵심 인사이트:** "비용-품질 사다리"의 중간이 비어 있다. $13(Canva)과 수천만원(전문팀) 사이에 $30-50로 FT급 품질을 얻을 수 있는 옵션이 없다.

## Target User & Narrowest Wedge

**1차 타겟:** KP 본인 — 프론트엔드 개발자이자 콘텐츠 크리에이터. AI 에이전트/법률 테크/투자 도구를 만들면서 그 과정과 인사이트를 고급 비주얼 스토리로 공유하고 싶은 사람.

**확장 타겟:** 텍스트 기반 콘텐츠를 만들고 있지만, 비주얼 스토리텔링으로 차별화하고 싶은 1인 미디어/뉴스레터 운영자.

**최소 웨지:** "토픽 하나를 입력하면 FT Visual Story 스타일의 완성된 인터랙티브 HTML 페이지가 나온다." 이 하나만 되면 가치 검증이 가능하다.

## Constraints

1. **Claude Code 환경:** Claude의 Task tool(서브에이전트), 파일 시스템, 웹 서치를 기반으로 동작
2. **HTML 산출물:** 최종 결과물은 self-contained HTML — 외부 CDN 의존 최소화, 오프라인에서도 동작
3. **Chart.js 기반 데이터 시각화 (D3 마이크로모듈은 선택적):** 표준 차트(bar, line, pie, doughnut)는 Chart.js, 커스텀 비주얼(timeline, treemap, force graph)은 D3 마이크로모듈만 사용. 실제 데이터가 없는 경우 시뮬레이션 데이터를 생성하되, 반드시 `[시뮬레이션 데이터]` 라벨을 차트 하단에 표시
4. **이미지 소싱 API 의존:** Unsplash/Pexels API로 실제 고품질 이미지 확보 (AI 생성은 최후 수단)
5. **품질 우선:** 시간/토큰 비용보다 결과물의 시각적 품질과 콘텐츠 깊이를 우선시
6. **한국어 우선:** MVP는 한국어 단일 언어. 영어 번역은 Phase 2 이후 지원
7. **이미지 전략:** 이미지는 `output/images/` 디렉토리에 별도 파일로 저장, HTML에서 상대 경로(`images/xxx.jpg`)로 참조. Self-contained 배포 시 `output/build/` 폴더 전체를 배포
8. **폰트 전략:** 시스템 폰트 스택 기본(`-apple-system, 'Pretendard Variable', sans-serif`). 온라인 환경에서는 Google Fonts CDN으로 Inter/Pretendard 로드하되, 오프라인 시 시스템 폰트로 graceful fallback
9. **성능 예산:** 최종 HTML < 500KB (이미지 제외). 이미지 포함 전체 번들 < 5MB. 이미지는 WebP 포맷, lazy loading 적용

## Premises

1. **FT Visual Story의 핵심 가치는 '본문 품질 + 데이터 시각화 + 시각 디자인'의 조합이다.** AI가 이 세 가지를 동시에 해낼 수 있다. *(합의됨)*
2. **타겟 유저(1인 미디어)가 월 $30-50 수준의 구독료를 낼 의지가 있다.** Shorthand($399/월)과 Canva($13/월) 사이의 가격대. *(합의됨, 검증 필요)*
3. **ebook-writer의 파이프라인 아키텍처 패턴을 재사용할 수 있다.** 직접 코드 포크가 아니라, 아키텍처 패턴(상태 관리, 에이전트 디스패치, 게이트, 웨이브 실행)을 새 파이프라인에 적용. *(합의됨)*

## Approaches Considered

### Approach A: ebook-writer Fork (최소 노력)
기존 ebook-writer 파이프라인을 분기해서 출력 포맷만 교체. Research/Outline/Write/Edit 에이전트 재사용, Writer에 '스크롤 섹션 단위' 프롬프트 추가, PDF 대신 scrollytelling HTML 출력.
- 노력: S (human: 1주 / CC: 2시간)
- 위험: Low
- 장점: 검증된 파이프라인 재사용, 가장 빠른 출시
- 단점: 책 구조에 갇힘 — 비주얼 스토리는 챕터가 아니라 시각적 beat로 흐름
- Completeness: 5/10 — 결국 한계에 부딪혀 재작성 필요

### Approach B: 전용 Visual Story Engine (이상적 아키텍처) ← SELECTED
새 파이프라인을 비주얼 스토리 전용으로 설계. 새 에이전트: Story Arc Designer, Visual Section Writer, DataViz Agent, Layout Assembler.
- 노력: L (human: 3주 / CC: 8-10시간, 2 Phase에 걸쳐)
- 위험: Med
- 장점: 비주얼 스토리에 최적화된 구조, 최고 품질 천장, DataViz 자동화
- 단점: 검증 전 투자가 크지만, CC로 Phase 1은 4-5시간이면 합리적
- Completeness: 7/10 (Open Questions 해소 후 9/10)

### Approach C: Template-First (측면 접근)
FT/NYT 비주얼 스토리를 분석해 5-10개 템플릿을 만들고 AI가 콘텐츠로 채우는 방식.
- 노력: M (human: 2주 / CC: 3시간)
- 위험: Low
- 장점: 예측 가능한 품질
- 단점: 템플릿 제약으로 창의성 제한, 모든 결과물이 비슷해 보임
- Completeness: 6/10

## Recommended Approach

**Approach B: 전용 Visual Story Engine**

**Phase 1 (MVP, CC 4-5시간):** Steps 1-4 + Step 7 — 리서치, 스토리 아크, 섹션 작성, 편집, HTML 조립. 데이터 시각화와 이미지 큐레이션은 수동 또는 간단한 스크립트로 대체.

**Phase 2 (Full, CC 4-5시간 추가):** Steps 5, 6, 8, 9 — DataViz 자동화, Image Curator, Polish Agent, 번역.

ebook-writer의 아키텍처 패턴(상태 관리, 게이트, 웨이브 실행)을 재활용하면서 비주얼 스토리에 최적화된 구조를 갖출 수 있다. Phase 1만으로도 수요 검증이 가능하고, "만들어서 본인이 써보는 것" 자체가 수요 검증 행위가 된다.

## Pipeline Architecture

```
[Input: Topic + Optional Data Sources]
    │
    ▼
  Step 1. Deep Research           → Research Agent
    │
    ▼
  Step 2. Story Arc Design        → Story Arc Designer Agent
    │       (visual beats, narrative flow, data points)
    ▼
  ══ Gate 1: Arc Approval ══      → Human review
    │
    ▼
  Step 3. Section Writing         → Visual Section Writer Agent (×N, parallel by wave)
    │       (scroll sections with visual directives)
    ▼
  Step 4. Editing & Fact-Check    → Editor Agent
    │
    ▼
  Step 5. Data Visualization      → DataViz Agent (NEW)
    │       (D3.js/Chart.js code generation)
    ▼
  Step 6. Image Curation           → Image Curator Agent (NEW)
    │       (real photos from Unsplash/Pexels, AI generation as last resort)
    ▼
  Step 7. HTML Assembly           → Layout Assembler Agent (NEW)
    │       (scrollytelling, responsive, interactive elements)
    ▼
  Step 8. Visual Polish & QA      → Polish Agent (NEW)
    │       (typography, spacing, animation timing, mobile)
    ▼
  Step 9. Translation (optional)  → Translator Agent (×N, parallel)
    │
    ▼
  ══ Gate 2: Final Review ══      → Human review
    │
    ▼
  [Deliverable: Self-contained interactive HTML]
```

## Sub-Agent Definitions

### 1. Research Agent (adapted from ebook-writer)
- **역할:** 토픽에 대한 심층 리서치 + 데이터 수집
- **변경점:** 데이터 시각화에 쓸 수 있는 정량 데이터/통계를 명시적으로 수집
- **산출물:** `output/research/research_report.md` + `data_points.json` + `citations.json`

### 2. Story Arc Designer Agent (NEW)
- **역할:** 리서치를 기반으로 비주얼 스토리의 내러티브 아크를 설계
- **핵심:** 챕터가 아니라 **visual beat** 단위로 설계
  - 각 beat: 스크롤 섹션 타입(hero, narrative, data-viz, quote, comparison, timeline, conclusion) + 예상 비주얼 + 핵심 메시지
- **산출물:** `output/arc/story_arc.json`
  ```json
  {
    "title": "...",
    "subtitle": "...",
    "total_beats": 12,
    "estimated_read_time": "8 min",
    "beats": [
      {
        "id": 1,
        "type": "hero",
        "headline": "...",
        "visual_directive": "Full-bleed image of ...",
        "data_viz": null,
        "transition": "fade-up"
      },
      {
        "id": 2,
        "type": "data-viz",
        "headline": "...",
        "visual_directive": "Animated bar chart showing ...",
        "data_viz": { "chart_type": "bar", "data_key": "market_growth" },
        "transition": "scroll-trigger"
      }
    ]
  }
  ```

### 3. Visual Section Writer Agent (NEW)
- **역할:** 각 beat의 본문을 작성 — FT/NYT 스타일의 간결하고 강렬한 프로즈
- **핵심:** 전통적 장문 글쓰기가 아니라, 스크롤 섹션에 맞는 짧고 임팩트 있는 텍스트
  - Hero: 1-2문장 훅
  - Narrative: 2-3 짧은 단락 (각 단락 2-3문장)
  - Data-viz: 데이터를 해석하는 인사이트 텍스트
  - Quote: 인용 + 맥락
- **산출물:** `output/sections/beat_{NN}_{type}.md` + inline visual directives

### 4. Editor Agent (adapted from ebook-writer)
- **역할:** 전체 스토리의 내러티브 일관성, 팩트체크, 톤 통일
- **변경점:** 책이 아니라 비주얼 스토리 관점에서 편집 — 섹션 간 전환 자연스러움, 중복 제거, 데이터 정확성
- **산출물:** 수정된 섹션 파일들 + `output/edit/edit_report.md`

### 5. DataViz Agent (NEW) — Phase 2
- **역할:** `data_points.json`과 각 beat의 `data_viz` 지시를 기반으로 Chart.js 또는 D3 마이크로모듈 코드 생성
- **라이브러리 선택 기준:**
  - **Chart.js** → 표준 차트: bar, line, pie, doughnut, radar, scatter
  - **D3 마이크로모듈** (`d3-scale`, `d3-shape`, `d3-axis` 등) → 커스텀: timeline, treemap, force graph, sankey
- **핵심:**
  - 각 차트를 self-contained JS 모듈로 생성
  - Scroll-trigger 애니메이션: **IntersectionObserver** 기본, CSS `scroll-timeline`은 progressive enhancement
  - 반응형 (모바일/데스크톱)
  - 접근성(ARIA labels, alt text)
  - 시뮬레이션 데이터 사용 시 `[시뮬레이션 데이터]` 라벨 필수
- **data_viz 스키마:**
  ```json
  {
    "chart_type": "bar | line | pie | doughnut | radar | scatter | timeline | treemap | force | sankey",
    "data_key": "research data_points.json의 키",
    "title": "차트 제목",
    "x_label": "X축 라벨 (해당 시)",
    "y_label": "Y축 라벨 (해당 시)",
    "animation": "fade-in | draw | count-up",
    "is_simulated": false,
    "color_scheme": "accent | neutral | sequential"
  }
  ```
- **산출물:** `output/dataviz/viz_{NN}_{type}.js` + `output/dataviz/dataviz_manifest.json`

### 6. Image Curator Agent (NEW — replaces ebook-writer's Image Generation)
- **역할:** 고품질 실제 이미지 확보 및 큐레이션 — AI 그림 생성이 아님
- **핵심 철학:** FT/NYT Visual Story는 AI 일러스트가 아니라 에디토리얼 포토그래피와 실제 데이터 시각화를 사용한다. 이 에이전트의 핵심은 **실제 쓸 수 있는 고품질 이미지를 확보하는 것**.
- **이미지 소싱 전략 (우선순위):**
  1. **무료 고품질 스톡:** Unsplash API, Pexels API — 에디토리얼 수준 사진
  2. **데이터 시각화 스크린샷:** DataViz Agent가 생성한 차트를 이미지로 캡처
  3. **공공 데이터/자료:** 정부 통계, 공개 보고서의 차트/그래프 (출처 명시)
  4. **사용자 제공 이미지:** `input/images/` 디렉토리에 사용자가 직접 넣은 이미지
  5. **AI 생성 (최후 수단):** 적합한 실제 이미지를 찾을 수 없는 경우에만 AI 이미지 생성 — 추상적 배경/텍스처 등 제한적 용도
- **이미지 품질 기준:**
  - 최소 해상도: 1920px wide (hero용), 1200px (inline용)
  - 라이선스: 상업적 사용 가능한 것만
  - 스타일 일관성: 토픽에 맞는 톤/무드 통일
- **산출물:** `output/images/` + `image_manifest.json` (각 이미지의 소스, 라이선스, 용도 기록)

### 7. Layout Assembler Agent (NEW)
- **역할:** 모든 콘텐츠(텍스트, 데이터비즈, 이미지)를 scrollytelling HTML로 **구조적으로 조립**
- **경계:** 구조적으로 올바른 HTML을 생성하는 것이 목표. 시각적 미세 조정은 Polish Agent의 역할
- **핵심:**
  - HTML + inline CSS + JS 조립 (이미지는 상대 경로 참조)
  - IntersectionObserver 기반 scroll-trigger 애니메이션 (CSS scroll-timeline은 progressive enhancement)
  - 반응형 디자인 (모바일 퍼스트)
  - 라이트 모드 단일 테마 (다크 모드는 Phase 2)
  - 읽기 진행률 표시
  - 목차 네비게이션
  - 소셜 공유 메타태그 (OG tags)
- **산출물:** `output/build/story.html` + `output/build/images/` (복사됨)

### 8. Polish Agent (NEW) — Phase 2
- **역할:** HTML 소스 코드 분석 기반 품질 검증 + 자동 수정
- **경계:** Layout Assembler가 만든 구조를 받아 코드 레벨에서 미세 조정. 시각적 렌더링 검증은 **headless browser 사용 가능 시에만** 수행.
- **코드 레벨 검증 (항상 수행):**
  - 타이포그래피: font-size 계층(최소 3단계), line-height, margin/padding 일관성
  - 접근성: img alt 속성, color contrast ratio (WCAG AA), 키보드 탭 순서
  - 성능: 이미지 lazy loading, JS defer, 전체 HTML 크기 < 500KB 검증
  - 반응형: media query 존재 여부, viewport meta tag
- **시각적 검증 (headless browser 가용 시):**
  - Puppeteer/Playwright로 데스크톱 + 모바일 스크린샷 캡처
  - 레이아웃 깨짐 확인
- **산출물:** 수정된 `output/build/story.html` + `output/qa/polish_report.md`

## Error Handling & Retry Strategy

각 단계별 최대 리트라이 2회. 실패 시 `output/logs/step_{N}_retry_{count}.md`에 로그 기록.

| Step | 주요 실패 모드 | 복구 전략 |
|------|--------------|----------|
| 1. Research | 웹 서치 실패, 데이터 부족 | 리트라이 → 토픽 범위 확장하여 재시도 |
| 2. Story Arc | 잘못된 JSON 생성 | 리트라이 → JSON validation 후 재시도 |
| 3. Section Writing | 특정 beat 작성 실패 | 해당 beat만 재시도 (다른 beat 유지) |
| 4. Editing | 편집 Agent 실패 | 리트라이 → 2회 실패 시 원본 유지, Gate 2에서 수동 확인 |
| 5. DataViz | 잘못된 JS 코드 생성 | 리트라이 → 2회 실패 시 static 이미지 대체 |
| 6. Image Curation | API rate limit, 적합한 이미지 없음 | 다른 소스로 폴백 → 최종 실패 시 placeholder |
| 7. HTML Assembly | HTML 생성 실패 | 리트라이 → 비-블로킹 (DataViz/Image 없이 텍스트 only 버전 생성 가능) |
| 8. Polish | 검증 실패 | 비-블로킹 — polish_report.md에 이슈 기록, 수동 수정 가능 |

**비-블로킹 원칙:** Image Curation과 DataViz 실패는 파이프라인을 중단시키지 않는다. 해당 요소가 빠진 상태로 HTML을 생성하고, 사용자가 Gate 2에서 수동으로 보완할 수 있다.

## Gate Interaction

**Gate 1 (Story Arc Approval):**
1. `output/arc/story_arc.json`을 읽어 터미널에 포맷된 요약 표시 (beat 목록, 타입, 예상 읽기 시간)
2. AskUserQuestion으로 선택지 제시: `approve` / `edit (피드백 제공)` / `reject (처음부터 재설계)`
3. 승인 시 → Step 3 진행, 피드백 시 → state.gate1_feedback에 저장 후 Step 2 재실행

**Gate 2 (Final Review):**
1. `output/build/story.html`을 브라우저로 열기 (`open output/build/story.html`)
2. 산출물 요약 표시 (beat 수, 이미지 수, DataViz 수, 파일 크기)
3. AskUserQuestion: `approve` / `revise (특정 beat 번호 + 피드백)` / `reject`
4. 승인 시 → 파이프라인 완료, 피드백 시 → 해당 beat만 Step 3부터 재실행

## Design System for Output

생성되는 비주얼 스토리는 일관된 디자인 시스템을 따른다:

- **Typography:** 한국어(Pretendard) + 영어(Inter) — 시스템 폰트 폴백
- **Color Palette:** 뉴트럴 베이스 + 1-2 액센트 컬러 (토픽에 따라 AI가 선택)
- **Layout Grid:** 12-column, max-width 1200px, 좁은 텍스트 컬럼(680px) + 풀블리드 비주얼
- **Spacing:** 8px 기반 그리드
- **Animation:** 절제된 scroll-trigger 애니메이션 — FT/NYT 수준의 우아함
- **Section Types:**
  | Type | Layout | Visual |
  |------|--------|--------|
  | Hero | Full-bleed image + overlay text | 시네마틱 이미지 |
  | Narrative | 좁은 텍스트 컬럼 + 사이드 이미지 | 서포팅 일러스트 |
  | Data-Viz | 풀 width 차트 + 인사이트 텍스트 | D3/Chart.js |
  | Quote | 센터 정렬 대형 텍스트 | 미니멀 |
  | Comparison | 2-column 비교 | 비주얼 카드 |
  | Timeline | 수직 타임라인 | 이벤트 마커 |
  | Conclusion | 좁은 텍스트 + CTA | 심플 |

## Quality Rubric ("FT급"의 조작적 정의)

생성된 비주얼 스토리가 "FT급"으로 인정받으려면 아래 체크리스트의 각 항목을 충족해야 한다:

| 영역 | 기준 | 측정 |
|------|------|------|
| **Typography** | 최소 3단계 폰트 크기 계층 (headline, subhead, body) | CSS 검사 |
| **Layout** | 텍스트 컬럼 max-width 680px, 비주얼 full-bleed 가능 | CSS 검사 |
| **Animation** | Scroll-trigger 애니메이션 ≥60fps, 절제된 사용 | 수동 확인 |
| **Data Viz** | 스토리당 최소 1개 인터랙티브 차트 | 카운트 |
| **Image** | Hero 이미지 ≥1920px, 실제 사진 (AI 아트 아님) | 해상도 검사 |
| **Mobile** | 모든 섹션 가로 스크롤 없이 읽기 가능 | viewport 320px 테스트 |
| **Performance** | 첫 로드 < 3초 (이미지 lazy loading 적용) | Lighthouse |
| **Content** | 각 beat 본문이 간결 (narrative beat 당 ≤150단어) | 워드 카운트 |
| **Narrative** | 스토리 아크에 hook → tension → insight → resolution 구조 | 수동 확인 |

## Success Criteria

1. **MVP 완성:** "토픽 입력 → 비주얼 스토리 HTML 출력" 파이프라인이 end-to-end로 동작

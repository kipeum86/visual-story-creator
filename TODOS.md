## Phase 1 (MVP)

### JSON Schema 정의
- **What:** story_arc.json, data_points.json, image_manifest.json에 대한 JSON Schema 정의 + 오케스트레이터에서 밸리데이션
- **Why:** Inter-agent 데이터 계약이 스키마 없이 암묵적이면, Story Arc Designer가 잘못된 JSON 생성 시 모든 하위 에이전트가 조용히 실패함
- **Pros:** cascading failure 방지, 디버깅 시간 대폭 절감, 에이전트 간 계약 명시화
- **Cons:** 스키마 정의 및 밸리데이션 로직 추가 작업 (CC: ~10분)
- **Context:** 오케스트레이터(CLAUDE.md)에서 각 단계 산출물을 다음 단계에 전달하기 전 스키마 검증. Python jsonschema 또는 JS ajv 사용. 실패 시 리트라이 로직과 연결
- **Depends on:** FT Analysis (beat 타입 확정 후 스키마 확정)

### Scrollama 통합
- **What:** Layout Assembler에서 raw IntersectionObserver 대신 Scrollama 라이브러리(~5KB) 사용
- **Why:** Scrollama는 step-based scrollytelling의 사실상 표준. resize 처리, fast scroll 대응, 모바일 quirks 등 edge case가 이미 해결되어 있음
- **Pros:** 검증된 라이브러리로 edge case 대응, 코드량 감소, 유지보수 용이
- **Cons:** 외부 의존성 추가 (~5KB, output/build/에 번들)
- **Context:** Layout Assembler 프롬프트에서 "Scrollama step-based architecture를 사용하여 scroll trigger 구현" 명시. scrollama.min.js를 output/build/js/에 포함
- **Depends on:** 없음

### Automated QA Script (Step 7.5)
- **What:** Layout Assembler 출력물(story.html)에 대한 자동 품질 검증 스크립트
- **Why:** Quality Rubric 9개 항목 중 6개를 자동화하면 매번 수동 확인 없이 품질 보장 가능
- **Pros:** 품질 회귀 방지, Gate 2 리뷰 부담 감소, CI 연동 가능
- **Cons:** 스크립트 작성 비용 (CC: ~15분)
- **Context:** 검증 항목: HTML validity, font-size 3단계 계층, text column ≤680px, img lazy loading + alt, viewport 320px overflow, HTML <500KB, beat당 ≤150 단어. Python 또는 Node.js 스크립트로 구현
- **Depends on:** Layout Assembler 완성 후

## Phase 2

### Auto-Deploy Step
- **What:** output/build/ 폴더를 GitHub Pages / Vercel / Netlify에 자동 배포하는 선택적 파이프라인 단계
- **Why:** "토픽 입력 → 발행까지" 완전 자동화. MVP에서는 수동 배포로 충분하지만, 주 1회 사용 목표 달성 시 자동화가 UX를 크게 개선
- **Pros:** 원커맨드 배포, 발행 마찰 제거
- **Cons:** 배포 플랫폼 선택 및 인증 설정 필요
- **Context:** GitHub Pages(`gh-pages`), Vercel(`vercel --prod`), Netlify(`netlify deploy`) 중 선택. Gate 2 승인 후 선택적 실행. pipeline_state.json에 deploy_url 기록
- **Depends on:** Phase 1 MVP 완성 + 수요 검증
